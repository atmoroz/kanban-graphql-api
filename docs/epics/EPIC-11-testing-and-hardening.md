## Epic 11: Testing & Hardening

**Мета:** Покрити ключові частини Kanban API тестами (юніт, інтеграційні, «сквозні» GraphQL-запити) та захистити API від некоректного вводу та спроб «поламати» логіку. Зафіксувати мінімальний, але реалістичний набір тест-кейсів.

---

## У scope

**Фічі:** базове тест-покриття для основних доменних сценаріїв та негативні кейси (валідація, права, некоректні дані).

**Типи тестів (мінімум):**

- **Unit-тести** для:
  - `lib/errors.ts`, `lib/pagination.ts`, `lib/auth.ts` (без реальних JWT-секретів).
  - Сервісів: `board.service`, `column.service`, `task.service`, `task-search.service`, `board-member.service`, `label.service`, `status.service`, `pending-invite.service`, `activity.service`.
- **Інтеграційні / GraphQL-тести**:
  - Запуск схеми в пам'яті (без реального HTTP) та виконання запитів/мутацій через GraphQL API.
  - Перевірка, що резолвери коректно використовують сервіси, права та формат помилок.

**Use-cases (позитивні):**

| ID | Дія |
|----|-----|
| UC-T1 | Успішний happy-path для кожної основної фічі: створити дошку, колонку, задачу; перемістити задачу; додати мітку; змінити статус; перевірити activity. |
| UC-T2 | Реєстрація/логін, отримання `me`, створення дошки від імені Owner, запрошення учасника, перевірка ролей. |
| UC-T3 | Email-інвайт по адресі, якого немає в системі: створюється PendingInvite, відправляється лист (у режимі `stub`), при реєстрації по цьому email інвайт застосовується. |

**Use-cases (негативні та хаос-кейси):**

| ID | Дія |
|----|-----|
| UC-NT1 | Спроба викликати мутації без токена (або з невалідним/простроченим токеном) → UNAUTHORIZED. |
| UC-NT2 | Спроба змінювати чужу дошку/колонку/задачу без достатньої ролі → FORBIDDEN. |
| UC-NT3 | Спроба звернутися до неіснуючих сутностей (board/column/task/user/status/label) → NOT_FOUND. |
| UC-NT4 | Невалідний ввід: порожні назви дошок/колонок/задач, занадто короткий пароль, некоректний email для інвайту, порожній список labelIds → VALIDATION_FAILED. |
| UC-NT5 | Конфлікти: повторна реєстрація з тим самим email, повторний inviteByEmail для тієї ж пари board+email, спроба додати учасника, який вже є → CONFLICT. |
| UC-NT6 | Спроби «поламати» фільтри/пошук: занадто великі ліміти, від'ємні ліміти, дивні cursor-значення, невідомі значення sortBy/TaskSortBy → коректна відмова або безпечний дефолт без падіння. |
| UC-NT7 | Спроба викликати підписки (realtime) без прав учасника дошки або без токена → UNAUTHORIZED/FORBIDDEN при subscribe. |

## Реалізація (high-level)

- **Тестовий раннер і структура:**
  - Обрати та зафіксувати тестовий фреймворк (наприклад Jest або Vitest) та команду `npm test`.
  - Структура: `src/__tests__/...` або `tests/` з вкладенням по доменах (`auth.test.ts`, `boards-columns.test.ts`, `tasks.test.ts`, `roles-permissions.test.ts`, `labels-statuses.test.ts`, `activity-log.test.ts`, `email-invites.test.ts`, `realtime.test.ts`).
- **GraphQL-інтеграція:**
  - Хелпери для виконання GraphQL-запитів/мутацій/підписок проти схеми (`schema`) в пам'яті.
  - Можливість підмінити контекст (currentUser, токен) для перевірки прав.
- **Ізоляція даних:**
  - Перед кожним тестом очищати in-memory моки (`boards`, `columns`, `tasks`, `users`, `boardMembers`, `labels`, `statuses`, `activities`, `pendingInvites`) або піднімати новий «тестовий світ».
  - Для email-інвайтів включати провайдер `stub`, щоб листи логувалися в консоль/буфер, а не йшли назовні.
- **Покриття негативних шляхів:**
  - Для кожного сервісу та резолвера — мінімум один тест на happy-path та 1–2 на помилкові шляхи (немає прав, невалідний ввід, неіснуючий id).

## Критерії прийняття

- [ ] Налаштовано тестовий раннер (скрипт `npm test`), тести запускаються локально та в CI (якщо є).
- [ ] Є базовий набір unit-тестів для `lib/*` та ключових сервісів (auth, boards, columns, tasks, roles, labels/statuses, invites, activity).
- [ ] Є інтеграційні GraphQL-тести для основних API-сценаріїв (створення/оновлення/видалення, переміщення, фільтри/пошук, підписки).
- [ ] Негативні сценарії (невалідний ввід, відсутність прав, неіснуючі сутності, погані параметри пагінації/сортування) покриті та повертають помилки у форматі Epic-01 (код + message).
- [ ] Для email-інвайтів є тести, що перевіряють: створення PendingInvite, відправку через stub-провайдер та застосування інвайтів при login/register.
- [ ] Підписки протестовані як мінімум на happy-path (підписка, потім мутація → подія отримана) та на відмову при відсутності прав.

## Борг / додаткові ідеї

- Додати property-based/фаззі-тести для найкритичніших ділянок (наприклад парсинг фільтрів та сортування, валідація вхідних даних) — випадкова генерація входів, перевірка, що API не падає та дає очікувані коди помилок.
- Розглянути snapshot-тести для GraphQL-схеми (SDL) та основних відповідей, щоб простіше відстежувати зміни контракту.
- Пізніше можна інтегрувати лінтери схеми/запитів (наприклад graphql-schema-linter) та перевірити, що помилки/коди/типи не розходяться з документацією.
