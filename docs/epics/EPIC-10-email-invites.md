# Epic 10: Email-based invites

**Мета:** Запрошення на дошку по email: створення інвайту, зберігання очікуючих запрошень, автоматичне прийняття при логіні/реєстрації (створення BoardMember або PendingInvite + відправка листа).

---

## У scope

**Фічі:** запрошення по email, pending invites, прийняття при вході/реєстрації.

**Use-cases:**

| ID | Дія |
|----|-----|
| UC-I1 | Запросити на дошку по email (inviteByEmail): викликаючий повинен мати право на дошці (наприклад OWNER/ADMIN). |
| UC-I2 | Список очікуючих запрошень по дошці (pendingInvites(boardId)) та по email (pendingInvitesByEmail(email)). |
| UC-I3 | При **логіні** або **реєстрації** по email: шукаємо запрошення по цьому email. |
| UC-I3a | Якщо користувач **вже є**: одразу створюємо BoardMember з роллю з інвайту, позначаємо інвайт прийнятим. |
| UC-I3b | Якщо користувача **немає**: створюємо PendingInvite (boardId, email, role, invitedBy, createdAt); опційно шлемо email (наприклад посилання на реєстрацію). |

**Логіка «прийняти при вході»:**

- Пошук користувача по email (при login — відомий email; при register — щойно вказаний email).
- Пошук усіх pending invites по цьому email.
- Для кожного інвайту: якщо користувач знайдений → створити BoardMember, інвайт виконано; інакше → залишити як PendingInvite та за потреби відправити email (або відкласти відправку до реалізації «справжньої» пошти).

**Реалізація:**

- Мок: сховище запрошень (наприклад `data/mock/invites.ts` або `pending-invites.ts`): сутність з полями boardId, email, role, invitedByUserId, createdAt, status (PENDING | ACCEPTED).
- GraphQL: тип `PendingInvite` (id, boardId, board?, email, role, invitedBy, createdAt). Mutation `inviteByEmail(boardId, email, role)`. Queries: `pendingInvites(boardId)`, за потреби `pendingInvitesByEmail(email)` (для «мої запрошення» по токену — по currentUser.email).
- Інтеграція з auth: після успішного login/register у сервісі (або в резолвері) викликати «застосувати інвайти по email»: знайти pending по email → для існуючого user створити BoardMember та оновити статус інвайту; для неіснуючого — залишити PendingInvite та відправити лист із запрошенням (див. нижче).

**Відправка email при інвайті (інтеграція з провайдером):**

- Підключити транзакційний email-сервіс, наприклад [Mailgun](https://www.mailgun.com/) (Email API для розробників: відправка, трекінг, валідація). Альтернативи: SendGrid, Resend, AWS SES — на вибір.
- При створенні PendingInvite (коли по email користувача ще немає в БД) — одразу відправляти лист запрошеному.
- **Текст листа** — короткий і зрозумілий, наприклад:
  - Заголовок: «Вас запрошено на дошку [назва дошки]»
  - Текст: «Вітаємо! [Ім'я того, хто запросив] запрошує вас до команди дошки «[назва дошки]» в Kanban Dashboard. Перейдіть за посиланням нижче, щоб приєднатися. Якщо у вас вже є акаунт — увійдіть; якщо ні — зареєструйтесь за цим же посиланням.»
  - Одне **посилання-кнопка**: залежно від того, чи є користувач в БД:
    - **Користувач є** → посилання веде на **сторінку входу** (login).
    - **Користувача немає** → посилання веде на **сторінку реєстрації** (register).
  - Базовий URL посилання (наприклад `https://app.example.com/login` та `https://app.example.com/register`) винести в конфіг (env); адресу замінити, коли буде відомий фронтовий домен.
- Передавати в лист: email запрошеного, назву дошки, роль, посилання (login або register). Ідемпотентність: не слати повторний лист по одному й тому ж PendingInvite, якщо вже відправляли (прапорець `emailSentAt` або аналог — за бажанням).

## Поза scope

- OAuth / SSO. Складні сценарії «прийняти інвайт по одноразовому посиланню» (можна залишити на потім).

## Критерії прийняття

- [ + ] Mutation `inviteByEmail(boardId, email, role)` створює запрошення; перевірка прав викликаючого на дошці (наприклад тільки OWNER/ADMIN можуть запрошувати).
- [ + ] Query `pendingInvites(boardId)` повертає список очікуючих запрошень по дошці (по email, без користувача в системі або ще не прийнятих).
- [ + ] При **логіні** по email: по цьому email знаходяться pending invites; для кожного створюється BoardMember (якщо користувач вже є), інвайт позначається прийнятим.
- [ + ] При **реєстрації** по email: той самий пошук по email; якщо користувач щойно створений — створюються BoardMember по всіх pending для цього email; інакше не застосовуємо.
- [ + ] Якщо по email користувача немає: створюється/залишається PendingInvite та **відправляється email** із запрошенням (через обраний сервіс, наприклад Mailgun).
- [ + ] Лист містить: привітальний текст, назву дошки, одне посилання — на **логін**, якщо користувач вже є в БД, або на **реєстрацію**, якщо немає. Базовий URL посилань у конфізі (env), щоб потім підставити реальну адресу фронту.
- [ + ] Тип `PendingInvite` у схемі (id, boardId, email, role, invitedBy, createdAt); за потреби зв'язок з Board.
- [ + ] Помилки у форматі Epic 1 (NOT_FOUND для дошки, FORBIDDEN при відсутності прав на запрошення).

---

## Борг

- **Посилання в листі:** при виклику `sendInviteEmail` з `pending-invite.service` не передається `actionUrl`. Підставити в лист посилання на реєстрацію (або логін), взявши базовий URL з env (наприклад `FRONTEND_REGISTER_URL` / `APP_INVITE_URL`) та передаючи його в `sendInviteEmail` як `actionUrl`.
- **pendingInvites(boardId):** зараз повертаються всі інвайти по дошці (PENDING та ACCEPTED). За потреби звузити до «очікуючих» — фільтрувати по `status === 'PENDING'` у сервісі або в моку.
